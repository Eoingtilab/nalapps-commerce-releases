# 날라앱스 커머스 릴리즈 절차

이 문서는 Core, 커머스용 애드온, 독립 플러그인의 공개 배포와 업데이트 기록 절차를 정의합니다.

## 1. 배포 전제

- 원본 소스는 각 Private 저장소에서 관리합니다.
- 공개 저장소에는 설치 ZIP·카탈로그·SHA-256·감사결과·릴리즈 노트만 게시합니다.
- app.nal.la EDD 브리지는 고객용 공개 자산에 포함하지 않습니다.
- 기능코드는 애드온형과 독립형이 하나의 저장소·하나의 버전·하나의 변경이력을 공유합니다.

## 2. 버전 규칙

Semantic Versioning을 사용합니다.

- Major: 호환되지 않는 구조·데이터·API 변경
- Minor: 하위 호환 신규 기능
- Patch: 하위 호환 버그·보안 수정
- 사전검증: `1.2.0-rc.1` 형식
- Git 태그·GitHub Release: `v1.2.0-rc.1` 형식

Core와 각 제품은 독립 버전을 가질 수 있으나, 같은 저장소에서 생성되는 커머스용·독립형 두 배포본은 항상 같은 기능 버전을 사용합니다.

## 3. 필수 릴리즈 게이트

### 공통

- PHP 전체 구문검사
- JavaScript 전체 구문검사
- 자동화 테스트 전체 통과
- 비밀정보·토큰·고객정보 정적 검사 0건
- ZIP 경로 순회와 비정상 루트 검사
- 두 번 독립 빌드한 산출물 SHA-256 일치
- 공개 패키지에 전체 소스 ZIP 미포함

### Core

- Core에 애드온 실행코드가 포함되지 않았는지 검사
- 최초 설치 마법사와 최소 1개 애드온 선택 검사
- WooCommerce 동의 설치 흐름 검사
- 애드온 ZIP 제품·폴더·메인 파일·Text Domain·버전·SHA-256 검사
- Core·애드온·Pro 라이선스 계약 검사

### 각 제품

- 커머스용 ZIP과 독립형 ZIP 모두 생성
- 두 ZIP이 동일 기능코드와 데이터 스키마 사용
- 동시 활성화 시 중복 실행 차단
- 제품별 라이선스·업데이트 상태 격리
- WooCommerce 요구 여부 정확성
- 비활성화·업데이트 시 데이터 보존

### 외부 인수시험

- 실제 WordPress·WooCommerce 스테이징 설치
- 기존 버전 업그레이드
- app.nal.la EDD 활성화·검증·만료·도메인 변경
- GitHub Release 자동 업데이트
- SMS·알림톡·SMTP 등 외부 제공업체 연동

외부 인수시험이 완료되지 않은 버전은 정식 Stable이 아니라 RC로 게시합니다.

## 4. 공개 자산

릴리즈마다 다음 자산을 게시합니다.

- Core 설치 ZIP
- 커머스용 애드온 ZIP
- 독립 플러그인 ZIP
- `nalapps-commerce-addons.json`
- `nalapps-standalone-products.json`
- 릴리즈 인벤토리 JSON
- SHA-256 검증파일
- 자동감사 결과 JSON
- 릴리즈 노트

## 5. 릴리즈 노트 필수 항목

- 버전·배포일·상태(Stable/RC/보안)
- 영향을 받는 제품과 배포형태
- 추가·변경·수정·보안 항목
- 요구 WordPress·PHP·WooCommerce·Core 버전
- DB 마이그레이션 여부와 데이터 보존 정책
- 라이선스·카탈로그·업데이트 계약 변경
- 알려진 제한사항
- 스테이징·자동검사 결과
- 각 자산 SHA-256
- 롤백 절차

## 6. 업데이트 게시 순서

1. 각 Private 저장소의 테스트와 변경이력을 완료합니다.
2. 태그 대상 커밋을 고정합니다.
3. 커머스용·독립형·EDD 운영용 패키지를 재현 빌드합니다.
4. 자동감사와 SHA-256을 생성합니다.
5. app.nal.la 카탈로그 호환성을 확인합니다.
6. GitHub Release를 Draft로 생성합니다.
7. 스테이징에서 Draft 자산을 설치·업데이트 시험합니다.
8. 결과를 릴리즈 노트에 기록합니다.
9. Release를 게시하고 원격 카탈로그를 갱신합니다.
10. WordPress 관리자 업데이트 알림과 다운로드를 재확인합니다.

## 7. 롤백

- 이전 정식 Release 자산을 삭제하지 않습니다.
- DB 변경은 가능하면 하위 호환·멱등 방식으로 설계합니다.
- 롤백 시 플러그인 파일만 이전 버전으로 교체하며 운영 데이터는 자동 삭제하지 않습니다.
- 되돌릴 수 없는 DB 변경은 사전 백업·복원 절차와 함께 릴리즈 노트에 명시합니다.
- 라이선스·카탈로그 장애 시 마지막 정상 카탈로그와 제한된 유예정책을 사용합니다.

## 8. 보안 릴리즈

- 취약점은 공개 Issue에 상세히 게시하지 않습니다.
- 수정·회귀검사·영향범위 확인 후 배포합니다.
- 릴리즈 노트에는 악용 가능한 상세 대신 영향버전·수정버전·필요 조치를 기록합니다.
- 비밀정보 유출 가능성이 있으면 관련 키와 토큰을 즉시 폐기·재발급합니다.

## 9. 기록 보존

- 저장소별 `CHANGELOG.md`: 제품 기능·버그·보안 변경
- 공개 `CHANGELOG.md`: 전체 배포 인덱스
- `releases/<version>.md`: 버전별 상세 릴리즈 기록
- GitHub Release 본문: 고객이 확인할 핵심 변경·설치·업데이트 정보
- 자동감사 JSON과 SHA 파일: 기계 검증 기록
